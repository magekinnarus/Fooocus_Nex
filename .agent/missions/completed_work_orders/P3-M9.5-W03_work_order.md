### `cast_to_device` Analysis
This is the most-called function. In `model_management.py` it's essentially:
```python
def cast_to_device(tensor, device, dtype, copy=False):
    # ... handles non_blocking, copy semantics
    return tensor.to(device=device, dtype=dtype, copy=copy)
```
The key subtlety is `non_blocking` for CUDA transfers. Must preserve this behavior exactly.

### `lora.py` Special Case
`lora.py` imports `model_management` but only uses it for `model_management.cast_to_device` indirectly through weight operations. After W02 absorbs key-mapping functions, verify that `lora.py` no longer needs direct `model_management` references.

## Risk Assessment

**Highest risk item: `unet_dtype()`**
This function in `model_management.py` has complex logic involving:
- Config flags (`fp32_unet`, `fp16_unet`, `bf16_unet`, etc.)
- Device capabilities (bf16 support check)
- Weight dtype inference
`resources.py` already has `should_use_fp16()` and `should_use_bf16()` which partially overlap. The extraction must unify this logic without changing behavior.

**Medium risk: `cast_to_device()` non_blocking semantics**
CUDA transfers can use `non_blocking=True` for overlapping compute and transfer. If the original uses this and our version doesn't, it could cause subtle performance differences.

**Low risk: `module_size()`**
Pure computation — sum of `param.nelement() * param.element_size()`. No side effects.
