## Objective

Extract three `ldm_patched.modules` infrastructure modules into native backend equivalents **before M10**, so that Fooocus integration work (ControlNet, inpainting, UI wiring) does not risk confounded debugging when regressions occur.

### Rationale

During M09-W04, a simultaneous LoRA implementation + `utils` extraction caused an undiagnosable performance regression, forcing a full rollback. M10 will add ControlNet, inpainting, and UI wiring — far more complex than LoRA. If infrastructure extraction happens during M10, the same two-variable debugging problem will be worse.

**Principle: extract infrastructure → verify → commit → then build features on clean foundation.**

## Scope

### Target Modules

| Module | Callsites | Primary Consumers | Risk |
|--------|-----------|-------------------|------|
| `ldm_patched.modules.float` | 4 calls | `weight_ops.py`, `patching.py` | Low — 68 lines, 3 pure functions |
| `ldm_patched.modules.utils` | ~10 calls | `patching.py`, `weight_ops.py`, `loader.py`, `lora.py` | Medium — `set_attr`/`copy_to_param` touch weight hot path |

## Constraints
- **One module per work order.** Never extract two simultaneously.
- **Each W must pass regression test before next W starts.**
- Regression test = SD1.5 + SDXL generation via `app.py` with identical seed/config, comparing: (a) output image similarity, (b) inference wall-clock time, (c) VRAM usage.
- Existing `ldm_patched/modules/` files are **not modified** — our backend simply stops importing from them.

## Success Criteria
1. Backend modules have zero imports from `float`, `utils`, `model_management`, or `comfy_types`
2. SD1.5 test config produces identical output (same seed → same image)
3. SDXL test config produces identical output
4. Inference time within ±5% of pre-extraction baseline (same config, 3-run average)
5. VRAM usage within ±5% of pre-extraction baseline
6. LoRA application still works correctly

## Notes
- W01 is the easiest — 68 lines, pure math, no side effects. Perfect confidence builder.
- W02 previously caused regression during M09-W04. The key functions to watch: `set_attr`/`copy_to_param` (weight placement), `load_torch_file` (checkpoint loading). This time, LoRA is already stable so any regression is clearly attributable.
- W03 is the highest-value extraction. `resources.py` already implements ~80% of what `model_management` provides. The core operation is `cast_to_device` (used ~35 times in `weight_ops.py` alone) — a simple `tensor.to(device, dtype)` wrapper. The risk is in `unet_dtype()` which has complex logic for selecting compute precision.
- `lora.py` uses `ldm_patched.modules.utils.unet_to_diffusers()` and similar key-mapping functions. These are LoRA-specific utilities that should be absorbed into `lora.py` itself or a shared key-mapping utility, not into the general `utils.py`.
